# FEAT-006: Trusted Time Source {#root}

## Простыми словами {#plain-language}

Раньше программа верила системным часам Windows. Поэтому достаточно было перевести часы на «день», чтобы ночная блокировка не срабатывала, даже если время так и оставить неверным. Теперь NightLock сам узнаёт настоящее время из интернета (NTP) и решает, ночь сейчас или день, по нему — а не по часам компьютера. Если кто-то переведёт часы, это ничего не изменит. Между обращениями к интернету программа отсчитывает время по внутреннему ходовому таймеру, который нельзя сдвинуть, меняя системное время.

## Goal {#goal}

Сделать так, чтобы решение «restricted vs unrestricted» (`FEAT-001`) основывалось на доверенном источнике времени, не подверженном изменению системных часов локальным пользователем. Это целевое «future hardening», предусмотренное в `spec://modules/core/FEAT-001-night-lock-window#clock-behavior`, и частичное закрытие «change the clock» из `spec://TECHDEBT#TD-001`.

## Scope {#scope}

Входит:
- получение настоящего UTC по SNTP (NTP) напрямую из приложения;
- внутренний монотонный (не зависящий от системных часов) отсчёт времени между синхронизациями;
- модель «якорь + монотонное смещение» как источник доверенного времени для оценки расписания;
- поведение при недоступности сети (fallback);
- логирование расхождения доверенного и системного времени как tamper-evidence;
- конфигурация серверов и периодичности синхронизации.

Не входит:
- защита от смены часового пояса Windows (см. `#open-questions`);
- криптографическая проверка ответа NTP (NTS/authenticated NTP);
- собственный сетевой стек/прокси; используется обычный UDP/123;
- изменение визуального поведения замка (`FEAT-001` остаётся как есть).

## Trusted Clock Model {#trusted-clock}

Доверенное время строится как «якорь плюс монотонное смещение»:

- При синхронизации приложение запрашивает UTC у NTP-сервера и сохраняет пару `(anchorUtc, anchorMonotonic)`, где `anchorMonotonic` — показание монотонного счётчика ОС (`Environment.TickCount64`), который растёт от момента загрузки и **не меняется** при правке системных часов.
- Доверенное текущее UTC = `anchorUtc + (monotonicNow - anchorMonotonic)`.
- Для оценки расписания (заданного в локальном времени суток) доверенное UTC переводится в локальное время через текущий часовой пояс Windows.
- Системные «настенные» часы (`DateTimeOffset.Now`) в расчёте restricted/unrestricted **не участвуют**, пока есть действующий якорь.

Свойства:
- перевод системных часов вперёд или назад не двигает доверенное время;
- между синхронизациями время идёт по монотонному счётчику, поэтому остаётся верным и без постоянного доступа к сети;
- сон/гибернация: монотонный счётчик ОС продолжает учитывать реальное прошедшее время после пробуждения (а при больших скачках следующая синхронизация всё выправит).

## Synchronization {#synchronization}

- Синхронизация выполняется при старте процесса и далее периодически (`ntpResyncMinutes`, по умолчанию 60).
- Список серверов берётся из конфигурации (`ntpServers`, по умолчанию `pool.ntp.org`, `time.windows.com`, `time.google.com`); сервера пробуются по порядку до первого успешного ответа.
- Один SNTP-запрос: UDP к порту 123, стандартный 48-байтовый пакет, чтение transmit timestamp из ответа; таймаут короткий (по умолчанию 3 секунды).
- Применять ответ только при базовой валидации (режим/стратум корректны, год правдоподобен) — иначе считать попытку неуспешной.
- Каждый процесс, оценивающий политику (служба и хелпер), держит собственный доверенный источник времени; общего IPC не требуется (согласно `spec://modules/core/INFRA-001-windows-runtime-baseline#runtime-architecture`).

## Fallback Behavior {#fallback}

- **Якорь уже получен, сеть пропала:** продолжать вести доверенное время по монотонному смещению; это нормальный режим, не ошибка.
- **Якоря ещё нет и NTP недоступен (например, оффлайн сразу после загрузки):** временно использовать системное локальное время как источник и пометить время как недоверенное (`IsTrusted = false`), продолжая попытки синхронизации с коротким интервалом. Это сохраняет работоспособность оффлайн ценой того, что предварительно сбитые часы на «холодном» оффлайн-старте до первой синхронизации не выправляются. Ограничение зафиксировано в `spec://TECHDEBT#TD-008`.
- При возвращении сети первая же успешная синхронизация устанавливает якорь и переводит время в доверенное.

## Tamper Evidence {#tamper-evidence}

- Когда доверенное время и системные часы расходятся больше, чем на порог (по умолчанию 2 минуты), записать в лог разовое предупреждение с величиной расхождения — это и есть фиксация попытки подмены часов.
- Расхождение **не** ломает работу и **не** уводит в постоянную блокировку само по себе; политика просто продолжает считать по доверенному времени.

## Resource Budget {#resource-budget}

- Источник времени не добавляет постоянной нагрузки: между синхронизациями он лишь читает монотонный счётчик (несколько арифметических операций на тик существующего таймера) — без отдельного потока-«крутилки».
- Сетевой запрос — короткий UDP раз в `ntpResyncMinutes`; вписывается в бюджет `spec://modules/core/INFRA-001-windows-runtime-baseline#resource-budget`.

## Config {#config}

Добавляемые поля конфигурации (`spec://modules/core/INFRA-001-windows-runtime-baseline#config`):
- `useTrustedTime` (bool, по умолчанию `true`) — использовать доверенный источник времени; `false` возвращает к чистым системным часам (поведение до `FEAT-006`).
- `ntpServers` (list of host, по умолчанию `["pool.ntp.org", "time.windows.com", "time.google.com"]`).
- `ntpResyncMinutes` (int, по умолчанию `60`, диапазон `5..1440`).
- `ntpTimeoutSeconds` (int, по умолчанию `3`, диапазон `1..15`).

Невалидные значения откатываются к умолчаниям (как у остальных полей `NightLockSettings`).

## Open Questions {#open-questions}

- Смена **часового пояса** Windows всё ещё сдвигает локальное отображение доверенного UTC и может сместить окно. Это отдельный вектор, вынесен в `spec://TECHDEBT#TD-009`; в `FEAT-006` не закрывается.
- Ответ NTP не аутентифицирован: теоретически подменяем при контроле над сетью. Для текущей модели угроз (`spec://TECHDEBT#TD-001`) приемлемо.

## Acceptance {#acceptance}

- При наличии интернета перевод системных часов (вперёд или назад, с возвратом или без) **не меняет** фазу политики: блокировка по-прежнему начинается и заканчивается по настоящему времени.
- После потери сети доверенное время продолжает идти корректно по монотонному счётчику (проверяемо в юнит-тестах с подставными часами).
- Расхождение системных и доверенных часов больше порога логируется один раз как tamper-evidence.
- При `useTrustedTime = false` поведение совпадает с прежним (системные часы).
- SNTP-запрос/разбор и модель «якорь + монотонное смещение» покрыты юнит-тестами без обращения к сети (инъекция монотонного счётчика и NTP-ответа).
